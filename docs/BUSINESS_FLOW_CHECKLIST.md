> 目的：沉淀当前已确认的业务规则、系统边界、实现映射和明显缺口，便于后续由 Claude Code 执行开发，由人工做产品验收与方向把关。

## 1. 协作约定

- 开发执行：优先由 Claude Code 完成代码实现、测试、基础 QA。
- 产品把关：由人工负责确认业务方向、验收结果、是否允许合并与上线。
- 合并与上线：默认不自动执行，除非人工明确确认“merge / 上线 / 部署”。
- 前端改动完成后，必须补一轮基于浏览器的 QA。
- 使用 Claude 时，优先鼓励它调用合适插件/能力，不要偷懒。
  - 例如：需要先梳理方案或基于仓库文件做计划时，优先使用类似 `Planning with files` 的能力。
  - 例如：需要走页面验收时，优先使用浏览器自动化能力。

## 3. 系统定性

- 系统定位：销售为主，运营为辅。
- 功能取舍优先级：
  - 先满足销售主管视角。
  - 再满足销售视角。
  - 最后满足运营视角。

## 4. 角色与主页面

### 4.1 销售主管 `sales-manager`

- 关注：商业机会、转化率、增长率、回款率。
- 职责：管理销售团队、分配规则、订单审批。
- 页面：`/dashboard`、`/manager/*`

### 4.2 销售 `sales`

- 关注：货源是否可分配、客户用量承诺、预设用量警告、回款提醒、个人年度目标进度。
- 职责：客户推进、跟进、发起新订单、补充本地销售信息。
- 页面：`/home`、`/customers`、`/follow-ups`

### 4.3 运营 `ops`

- 关注：账单中心、客户详细用量、货源使用金额。
- 职责：从账单和用量角度回答“谁在哪个货源用了多少、花了多少钱”。
- 页面：`/bills`、客户详情中的用量相关视图

## 5. 核心业务主线

### 5.1 客户主线

1. 销售录入客户，或在新建客户时一步带出新订单流程。
2. 客户在销售系统中经历线索/沟通/推进过程。
3. 当订单被销售主管审批通过后，客户还不能直接视为正式客户。
4. 审批通过后，销售需要到工单系统补录客户编号。
5. 工单系统录入并同步回销售系统后，该客户才自动转成正式客户。

### 5.2 订单主线

1. 新建订单入口在客户管理，不在订单管理页。
2. 新建订单必须上传合同文件。
3. 新建订单支持多个货源。
4. 新建订单先进入审批，不即时生效。
5. 审批通过后，才进入后续资源/交付/账单链路。

### 5.3 货源主线

1. 货源主数据来自 cloudcost。
2. 货源状态以云管为准。
3. 看板按 `status` 聚合展示，不展示本地硬凑的配额数字。

### 5.4 用量与账单主线

1. 客户和货源的实际费用下钻依赖 `cc_usage`。
2. 月账单以云厂商汇总粒度为主。
3. 运营查看“某客户在哪些货源用了多少”时，应从 `cc_usage` 下钻。
4. 账单中心包含折扣计算器和导出能力。

### 5.5 AI 洞察主线

1. AI 洞察不只是公网搜索。
2. 必须综合本地销售数据：客户档案、跟进记录、合同、历史订单等。
3. 洞察结果持久化，后续增量更新。

## 6. 工单系统边界

这是当前业务里最重要的外部系统边界之一。

### 6.1 已确认规则

- 公司更早落地的是工单系统，所以客户编号当前保存在工单系统。
- 销售系统里的“正式客户”状态，当前以工单系统为准。
- 销售主管审批订单通过后，销售仍需到工单系统增加客户编号。
- 工单系统客户信息同步回销售系统后，客户才自动转成正式客户。
- 当前阶段不会把客户编号主源头迁回销售系统，但未来可能调整。

### 6.2 必须守住的业务一致性

- 不能出现两个相同客户。
- 不能出现工单系统和销售系统客户状态不一致。
- 订单审批通过和“正式客户生效”必须是两个阶段，不能混为一步。

### 6.3 对实现的含义

- 客户状态机必须能表达“审批已通过，但尚未被工单系统正式建号”的中间态。
- 正式客户状态的切换，应由工单同步触发，而不是仅由本地审批触发。
- 同步策略必须优先处理去重和状态统一，避免编号冲突。

## 9. 明显缺口与待核点

以下项目需要后续逐条核对，不宜直接假设已完全正确。

### 9.1 状态机缺口

- 需要确认当前模型和接口是否真正区分了：
  - 订单审批通过
  - 工单系统建号完成
  - 正式客户生效
- 若没有明确中间态，容易导致“审批一通过就变正式客户”的错误实现。

### 9.2 去重策略缺口

- 目前看到的是基于 `customer_code` 冲突的处理。
- 仍需确认以下情况是否处理完善：
  - 尚未建号时，销售先手工建了潜客，后来工单同步回来的是同名客户。
  - 同一客户在销售系统和工单系统名称略有差异。
  - 渠道客户、转介绍客户、直客在同步时的合并口径是否统一。

### 9.3 跨系统状态统一缺口

- 需要明确工单系统同步是定时、手动触发还是事件驱动。
- 需要确认工单系统回写失败时，销售系统如何保持“待正式化”状态，而不是脏切换。

### 9.4 发布风险

- CI 仍需继续向“在 Docker 中执行”收敛。
- 生产环境 schema 与主分支迁移需持续核对，避免上线后状态流转相关接口 5xx。
