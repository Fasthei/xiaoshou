# 销售系统产品方案

## 1. 项目背景
当前公司内部的云管系统、工单系统、客户交付表、客户表、CRM 等信息分散在多个系统中，且不同角色（销售、运营、配货人员）无法直接登录或查看完整信息，导致销售侧在客户跟进、货源分配、成本判断、价格核算、账单生成、用量预警等环节存在明显断层。

因此需要建设一个新的“销售系统”，但它本质上不是单纯的销售录单系统，而是一个面向销售主流程、兼顾运营协同、财务核算和智能分析的业务中台。

该系统的目标是：
- 让销售在一个系统内完成客户识别、货源查看、资源分配、价格判断、用量跟踪、账单处理、商机管理。
- 让运营和配货相关角色在权限范围内协同工作。
- 让管理层看到销售过程、资源消耗、潜在客户和收入预测。
- 通过 Agent / API 能力逐步形成“运营大脑”。

---

## 2. 产品定位
销售系统定位为：

**一个以销售主流程为核心，融合客户管理、货源分配、成本价格可见、用量分析、账单结算、商机评估与 Agent 自动化能力的综合业务平台。**

它需要覆盖：
- 销售管理
- 客户管理（CRM）
- 货源分配管理
- 用量与预警分析
- 费用与账单能力
- Agent 驱动的数据补全与商机判断
- API 化能力输出

---

## 3. 现阶段核心问题

### 3.1 信息割裂
- 客户基础信息在工单系统
- 货源与库存信息在云管系统
- 客户交付信息在交付表
- 客户跟进与销售过程在 CRM 或线下表格
- 费用、折扣、账单逻辑分散在人工流程中

### 3.2 角色权限导致关键数据不可见
- 销售和运营无法直接登录云管系统，看不到完整成本与货源状态
- 配货人员无法登录销售系统，看不到出货价格
- 不同岗位拿不到完成业务决策所需的完整数据

### 3.3 销售流程缺少业务闭环
当前缺少一套从：
- 客户识别
- 货源选择
- 资源分配
- 成本判断
- 价格核算
- 用量跟踪
- 风险预警
- 账单生成
- 客户续费/扩容
形成的统一闭环。

### 3.4 潜在客户与客户数据沉淀不足
- 客户资料补全依赖人工
- 潜在客户识别效率低
- 商机判断依赖个人经验，缺少系统化能力

---

## 4. 产品目标

### 4.1 业务目标
- 建立销售主流程统一入口
- 打通工单系统、云管系统、交付表、客户表、CRM
- 支撑客户、货源、成本、价格、消耗、账单的一体化管理
- 提升销售跟进效率、资源分配效率和账单处理效率

### 4.2 管理目标
- 让管理层掌握客户、销售、库存、分配、收入、风险全局情况
- 支撑潜在商机识别和客户价值分层
- 支撑收入预测和资源规划

### 4.3 智能化目标
- 利用 Agent 自动补全客户资料
- 利用 Agent 评估潜在成单用户和潜在客户
- 利用 Agent 进行用量预警、续费提醒和经营分析
- 最终沉淀为“运营大脑”能力，并可通过 API 对外提供能力

---

## 5. 系统边界与产品范围

### 5.1 本系统负责
- 客户主数据整合
- 客户 CRM 管理
- 货源展示与分配
- 成本与价格可见性补位
- 用量查询、趋势预测、预警
- 折扣与账单生成
- 潜客与商机分析
- Agent 自动化能力接入
- API 能力开放

### 5.2 外部系统提供
- 工单系统：客户编号、客户名称、基础工单信息
- 云管系统：货源、库存、账号、资源定义、可分配状态、用量/消耗原始数据
- 客户交付表：交付关系、交付记录、客户资源对应关系
- CRM / 客户表：历史客户信息、销售跟进记录（如已有）

## 6. 目标用户与角色

### 6.1 销售
关注：
- 客户是谁
- 当前有没有可分配货源
- 分配给客户后是否赚钱
- 客户当前用了多少
- 未来是否会超量/续费/扩容
- 如何快速报价、折扣、出账单

### 6.2 运营
关注：
- 客户交付情况
- 分配执行状态
- 资源可用情况
- 客户风险和预警

### 6.3 配货人员
关注：
- 分配指令
- 对应客户
- 对应货源
- 需要可见出货价格或至少可见与执行相关的重要价格信息

### 6.4 财务/商务
关注：
- 成本
- 折扣
- 账单
- 收款/应收
- 对账逻辑

### 6.5 管理层
关注：
- 销售漏斗
- 客户活跃与消耗
- 货源利用率
- 收入与利润趋势
- 潜客转化预测

## 7. 核心业务流程

### 7.1 客户识别与建档
1. 从工单系统同步客户编号、客户名称
2. 建立销售系统客户主档
3. 关联交付表、客户表、CRM 既有数据
4. Agent 自动补充行业、联系人、使用场景、历史需求等信息

### 7.2 货源查看与分配
1. 从云管系统同步货源池
2. 区分原厂货源与其他货源
3. 销售查看可分配资源和当前状态
4. 选择货源并分配给客户
5. 记录分配结果、分配人、时间、价格、成本、备注

### 7.3 分配后客户用量跟踪
1. 拉取客户当前消耗
2. 展示历史用量趋势
3. 预测未来用量变化
4. 触发预警（低余量、高消耗、异常增长、即将超配）

### 7.4 费用结算与账单
1. 根据货源、客户、分配关系、消耗数据计算费用
2. 支持折扣规则配置
3. 快速生成报价单/账单
4. 支持人工校正和审核

### 7.5 CRM 跟进与商机分析
1. 记录销售跟进动作
2. Agent 自动收集客户公开信息和历史记录
3. 对潜在客户做成单概率评估
4. 对现有客户做扩容/续费/流失风险提示

## 8. 核心功能模块

## 8.1 客户中心
功能包括：
- 客户主档
- 客户编号/客户名称同步
- 联系人管理
- 客户标签
- 客户级别/行业/地区
- 客户当前状态
- 客户历史交付与资源关联

### 关键字段建议
- 客户编号
- 客户名称
- 客户简称
- 所属销售
- 所属运营
- 行业
- 联系人
- 联系方式
- 客户状态
- 首次成交时间
- 最近跟进时间
- 当前在用资源数
- 当前月消耗
- 预计下月消耗

---

## 8.2 货源中心
功能包括：
- 货源池展示
- 原厂货源 / 其他货源分类
- 货源可用状态
- 分配状态
- 货源成本展示
- 货源标识规则管理

### 货源识别规则
#### 其他货源
- 以 key 前 6 位作为区别标识

#### 原厂货源
- AWS：账号名称 / 定义名称 / AWS 账号 ACID
- Azure：账号名称 / 定义名称
- GCP：账号名称 / 定义名称

### 关键字段建议
- 货源ID
- 货源类型
- 云厂商
- 标识字段
- 可分配数量
- 已分配数量
- 成本
- 建议销售价
- 状态
- 最近同步时间

---

## 8.3 分配管理
功能包括：
- 销售选择货源分配给客户
- 查看客户已有分配情况
- 查看货源分配历史
- 查看分配价格 / 成本 / 毛利
- 配货执行状态跟踪

### 核心要求
- 销售可看到货源情况和可分配情况
- 分配后自动同步客户维度视图
- 分配和交付信息联动
- 重要价格信息对相关角色可见

---

## 8.4 用量与预警中心
功能包括：
- 当前用量查询
- 历史趋势分析
- 用量预测
- 客户级预警
- 货源级预警
- 异常消耗识别

### 预警类型建议
- 剩余额度不足
- 用量异常增长
- 预计未来 7 天 / 30 天超阈值
- 已分配但长期未使用
- 低毛利客户预警

---

## 8.5 CRM 与商机中心
功能包括：
- 客户跟进记录
- 商机阶段管理
- 潜客池
- 成单概率评估
- 客户画像自动补全
- 自动提醒销售跟进

### Agent 能力建议
- 自动抓取并整理客户基础信息
- 自动总结沟通记录
- 自动打标签（行业、规模、需求、紧急度）
- 自动识别高潜客户
- 自动识别沉默客户、流失风险客户

---

## 8.6 财务与账单中心
功能包括：
- 成本查看
- 快速折扣计算
- 快速报价
- 快速生成账单
- 客户账期管理
- 毛利核算

### 必须解决的问题
- 销售不能看不到成本
- 配货人员不能看不到出货价格
- 财务不能依赖手工拼接分配和消耗数据

### 建议能力
- 按客户自动汇总费用
- 按货源自动汇总成本
- 支持折扣模板
- 支持手动调整价格
- 支持账单导出

---

## 8.7 运营大脑 / API 平台
功能包括：
- 对外提供统一 API
- 对内提供 Agent 调度能力
- 聚合客户、货源、分配、消耗、账单、商机数据
- 输出分析能力给其他系统

### API 能力建议
- 客户查询 API
- 货源查询 API
- 分配关系 API
- 用量查询 API
- 账单查询 API
- 预警查询 API
- 潜客评分 API

这部分是未来形成“运营大脑”的关键底座。

## 12. 产品一期建议范围（MVP）
优先实现：
1. 客户主档同步（工单系统）
2. 货源同步与展示（云管系统）
3. 货源分配给客户
4. 客户当前消耗查询
5. 成本/价格可见性补位
6. 基础 CRM 客户表
7. 快速折扣/报价/账单
8. 基础预警

### 一期目标
先把“销售能闭环完成客户识别、货源分配、价格判断、消耗查看、账单输出”跑通。

---

## 13. 二期建议范围
- 用量趋势预测
- Agent 自动补全客户资料
- 潜客评分与成单概率评估
- API 平台化输出
- 经营分析看板
- 更细颗粒度权限体系

---

## 14. 成功标准
如果系统上线后达到以下效果，则说明产品方向正确：
- 销售不再依赖多个系统来拼客户和货源信息
- 销售能直接判断资源是否可分配、是否赚钱
- 客户当前用量和风险可快速查看
- 报价、折扣、账单效率明显提高
- 潜客识别和客户跟进效率提升
- 管理层可看到业务全局与趋势

---

## 15. 待确认问题
当前仍需业务侧进一步确认：
1. 销售、运营、配货、财务各角色的精细权限边界
2. 云管系统能提供哪些成本/库存/消耗字段
3. 折扣规则与账单规则是否统一
4. 交付表、客户表、CRM 是否已有标准数据源
5. Agent 可使用哪些内部/外部数据源
6. API 是否只对内开放，还是未来对外开放

---

## 16. 最终结论
这个“销售系统”本质上不是单纯销售录单工具，而是一个：

**以销售主流程为核心，整合客户、货源、成本、价格、消耗、账单、CRM、Agent 与 API 能力的综合业务平台。**

它要解决的不是单点功能，而是公司当前销售、运营、配货、财务之间因为系统割裂和权限割裂带来的业务断层问题。


销售权限:权限、主管、销售
商机：商机来源，分配给谁，销售没时间在分配
